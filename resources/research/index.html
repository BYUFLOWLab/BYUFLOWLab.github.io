Research

Data Management

  • Use a lab notebook.
  • Version control is important. We use version control to develop code, to write papers, to collaborate, to rollback changes, etc. Use good commit statements so that you have a useful version history. Version control is helpful even on individual projects.
  • Archive results in addition to the code that produced it. Also archive the code that you used to create plots. This will save you a lot of time when regenerating plots. Plot regeneration is often needed to create versions of your plots for your presentation, or in responding to peer-reviewer's suggestions on your publications.
  • Backup. You need to have your data, code, etc. backed-up in at least two separate locations (in addition to its main location).

Software

  • Most of us are using Python (v2) as our primary programming language. We also use Matlab, Julia, Fortran, and C/C++. There is a useful online course on High Performance Scientific Computing that you can access for free: (Coursera course, course notes). It covers many relevant topics like version control, Unix commands, Python, Fortran, and MPI. These are all topics that will be useful to you.
  • We use Git for version control. Become comfortable with the basic commands: init, clone, add, commit, checkout, status, log, push, pull. We use Bitbucket to host our repositories. If you are actively working in our research group, sign up for a Bitbucket account and send me your username so that I can add you to our team.
  • For backup I use Dropbox (continuous cloud-based backup) and Time Machine (local hourly backup). You can also use BYU's J Drive if you like. There is more space there, and it is backed up, but restoration of files is more cumbersome.
  • We use LaTeX for technical writing. All of our publications are written using LaTeX.
  • For those interested in computational fluid dynamics, we are primarily using StarCCM+. There are good tutorials available on the CAEDM computers.
  • My text editor of choice is Sublime Text, but depending on the programming language you are using, you may prefer a dedicated IDE.
  • For graduate students in our lab, there are instructions in the setup repo for tools to download and configurations settings for your computer.

Code Development

  • Document your code in detail as you go. It may feel cumbersome at times, but it will save you time in the long run.
  • Use descriptive variable names
  • If you find yourself writing a very similar piece of code more than once, you should refactor the code to create one function used in multiple places. Having duplicate code opens up additional opportunities for bugs with version mismatches.
  • Use version control.
  • Create unit tests as you go. You will need to check the validity of your code anyway. When you perform those checks encapsulate them in unit tests. As you update code be sure to update your unit tests as well.

Productivity

  • Eliminate distractions (e.g., email, facebook, etc.). Don't try to "multitask". Research studies suggests it can take up to 20 minutes to get back into your task once you are distracted. At the same time you should also take planned breaks. Those are the times to check email, etc. You can get a lot more done with focused periods of works followed by short breaks, as compared to longer sessions with distractions. The same applies generally in life—try to be present with the people you are with.
  • Set and enforce your own deadlines. If you're not careful, work will expand to fill the time allotted.
  • Tackle hard things first, don't put them off. At the same time don't try to tackle the hard things in marathon sessions. First, you'll have a harder time finding a huge block of contiguous time. Second, you'll get burned out and procrastinate starting it up again. You'll be amazed at what you can accomplish with short but consistent efforts.
  • Be results oriented. It is easy in research to let irrelevant pursuits consume too much attention (because they are easier).
  • Network. Most people underestimate the importance of networking. It is vitally important. Get to know the other students and faculty members around you. Don't be afraid to email or approach people whose names are on the research papers you are reading. More often than not they will be flattered to talk to you about their work. At the same time, look for opportunities to mentor and help others.
