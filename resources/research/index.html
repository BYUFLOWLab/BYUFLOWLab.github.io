Research

Data Management

  • Use a lab notebook. This can be a hard bound book or a digital notebook (i.e., Jupyter).
  • Version control is important. We use version control to develop code, to write papers, to collaborate, to rollback changes, etc. Use good commit statements so that you have a useful version history. Version control is helpful even on individual projects.
  • Archive results in addition to the code that produced it. Also archive the code that you used to create plots. This will save you a lot of time when regenerating plots. Plot regeneration is often needed to create versions of your plots for your presentation, or in responding to peer-reviewer’s suggestions on your publications.
  • Backup. You need to have your data, code, etc. backed-up in at least two separate locations (in addition to its main location). This is a must! At some point you will lose data if you do not have backups.

Software

  • Julia and Python (v2) are our primary programming language. We also use Matlab, Fortran, and C/C++. Here are some learning resources:
    • High Performance Scientific Computing Coursera course, course notes, this course covers many relevant topics like version control, Unix commands, Python, Fortran, and MPI
    • Learn Python the Hard Way
    • Python 2.7 Docs
    • Julia Docs
  • We use Git for version control. Become comfortable with the basic commands: init, clone, add, commit, checkout, status, log, push, pull. We use GitHub to host our repositories. If you are actively working in our research group, sign up for a GitHub account and send me your username so that I can add you to our team. Our naming convention for repositories is all lowercase. Avoid separators where possible, but if it will help readability use a dash to separate words, not an underscore.
  • For backup I use Dropbox (continuous cloud-based backup) and Time Machine (local hourly backup). You may want to store larger files on BYU’s J Drive. There is more space there, and it is backed up, but restoration of files is more cumbersome. We also have university subscriptions to CrashPlan.
  • We use LaTeX for technical writing. All of our publications are written using LaTeX.
  • For those interested in computational fluid dynamics, we are primarily using StarCCM+ and SU2. For the former, there are good tutorials available on the CAEDM computers, and I teach quite a bit on its usage in ME 412.
  • My current text editor of choice is Atom, but depending on the programming language you are using, you may prefer a dedicated IDE. Some of the main packages I use include: script, uber-juno, latex, and zen.
  • Become comfortable using the Command Line.
    • Code Academy Command Line
    • Command Line Ninja
  • For graduate students in our lab, there are instructions on our wiki for tools to download and configurations settings for your computer.

Code Development

  • Comment your code in detail as you go. It may feel cumbersome at times, but it will save you time in the long run. Also provide some documentation, at least a README. Having details of a derivation will be really helpful to you later.
  • Use descriptive variable names
  • If you find yourself writing a very similar piece of code more than once, you should refactor the code to create one function used in multiple places. Having duplicate code opens up additional opportunities for bugs with version mismatches.
  • Use version control.
  • Create unit tests as you go. You will need to check the validity of your code anyway. When you perform those checks encapsulate them in unit tests. As you update code be sure to update your unit tests as well.
  • IT 515R, Scientific Computing, would be a good class for you to take.
  • Do code reviews, just like we do with paper reviews. In other words, when a model is more or less complete, ask a couple peers to review and offer feedback on correctness as well as clarity and style. Then let me know and I’ll review the code as well. Sometimes a hard to find bug or missing concept goes unnoticed and ends up surfacing us much later when we are putting results together. Frequent code reviews can help alleviate this issue.
